# Evaluation

The benchmark evaluates diagnostic quality, not only whether a compiler rejects bad code.

## Test Cases

The repository includes eleven valid programs:

| File | Language | Focus |
|---|---|---|
| `testcases/c/basic_mixed.c` | C | functions, arrays, pointers, structs, format output |
| `testcases/c/functions_arrays.c` | C | function calls and arrays |
| `testcases/c/pointers_structs.c` | C | pointers and struct fields |
| `testcases/c/control_flow.c` | C | loops and conditionals |
| `testcases/c/strings_memory.c` | C | strings, struct arrays, pointer arguments, format output |
| `testcases/c/preprocessor_typedefs.c` | C | macros, typedef structs, pointer updates, while loops |
| `testcases/fortran/basic_subroutine.f90` | Fortran | program, subroutine, arrays, intrinsic |
| `testcases/fortran/arrays_control.f90` | Fortran | arrays, loops, IF blocks |
| `testcases/fortran/procedures_intrinsics.f90` | Fortran | subroutine arguments and intrinsics |
| `testcases/fortran/modules_kinds.f90` | Fortran | module use, contained subroutine, arrays, intrinsics |
| `testcases/fortran/io_select_case.f90` | Fortran | SELECT CASE, logicals, internal subroutine, formatted output |

This satisfies the requirement of at least five test cases and gives the mutation engine more varied source patterns.

## Metrics

Each injected mutation is scored using six metrics.

| Metric | What it measures |
|---|---|
| Diagnosis accuracy | Whether the compiler identifies the right kind of mistake |
| Location precision | Whether the reported line is close to the injected line |
| Severity correctness | Whether the diagnostic is an error or warning as expected |
| Fix-it usefulness | Whether the compiler gives a useful repair hint |
| Explanation clarity | Whether the message is understandable and specific |
| Signal-to-noise | Whether the useful diagnostic appears early and is not drowned by cascades |

The final score is a fuzzy score from 0 to 100. Fix-it suggestions are evaluated more strictly when the compiler emits structured fix-it ranges: the suggested source range must be near the injected mutation line to receive strong credit.

| Assignment requirement | Covered by |

## Baseline Comparison

A simple baseline would compare compilers using only:

- did compilation fail?
- how many errors were printed?
- how many warnings were printed?

That baseline is weak because a compiler can print many errors and still not explain the actual mistake well.

This project improves on the baseline by checking whether the diagnostic matches the known injected error. Since the benchmark knows exactly what mistake was inserted, it can measure location and explanation quality directly.

## Output Files

After running:

```bash
./run.sh
```

the project writes:

```text
reports/benchmark_report.md
reports/benchmark_report.json
reports/benchmark_scores.csv
```

The Markdown report is meant for reading. The JSON report is meant for detailed inspection. The CSV report is useful for tables, charts, and spreadsheet analysis.

## How to Read the Score

| Score Range | Grade |
|---:|---|
| 85-100 | excellent |
| 70-84 | good |
| 50-69 | fair |
| 30-49 | weak |
| 0-29 | poor |

Examples:

- A high score means the compiler found the right problem, near the right line, and explained it clearly.
- A medium score means the diagnostic was useful but incomplete.
- A low score means the diagnostic was missing, misleading, too vague, or buried under unrelated output.

## Weak Spot Analysis

The benchmark also reports weak categories for each compiler.

For example, if a compiler performs poorly on scope errors, the report may suggest:

```text
Include the unresolved identifier and nearest similar declarations when possible.
```

If it performs poorly on format-string errors, the report may suggest:

```text
Pair each format specifier with the actual argument type.
```

These suggestions make the report more useful than a leaderboard. The point is not only to say which compiler scored higher, but also to show how diagnostics could improve.

## Expected Linux Evaluation Process

On a Linux machine:

```bash
chmod +x build.sh run.sh
./build.sh
./run.sh
```

If all compilers are installed, the report compares GCC, Clang, gfortran, and Flang. If one is missing, that absence is recorded in the report so the evaluator can see the environment clearly.
